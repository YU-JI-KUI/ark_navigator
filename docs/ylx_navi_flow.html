三、6 种最终响应

★ 响应 #1 - 异常兜底

触发条件：任何步骤抛未捕获异常（如小安访问失败、KnowledgeInfo 构造失败等）
code=-1, source_bu_type=ylXian, card_type=ylXian, service_type=rejection, card_content={}

★ 响应 #2 - 意图非养老险

触发条件：意图识别结果不是"养老险意图"（如"紧急救援"、"拒识"等）
code=0, source_bu_type=ylXian, card_type=ylXian, service_type=rejection, card_content={}

★ 响应 #3 - 非 ylXian 渠道

触发条件：意图是养老险但渠道不是 ylXian，直接返回小安原始数据
code=0, source_bu_type=ylXian, card_type=ylXian, service_type="", card_content=小安data, source=ylXian

★ 响应 #4 - LLM 判定"其他"

触发条件：大模型识别 sub_intent 为"其他"或 domain 为"其他"
card_content=小安data, source=ylXian

★ 响应 #5a / #5b - 一键流程中途退出

触发条件：
  • 5a: Table 知识库查不到对应 sub_intent
  • 5b: task_type=="Info" 且小安 answerType=="7"（动态菜单）
card_content=小安data, source=ylXian

★ 响应 #6 - 一键卡片命中（核心场景）

触发条件：所有检查通过，进入一键场景
code=0, card_type=onekey, source=onekey, card_content=Table 知识 + 小安 answer 拼装

六、流程中发现的问题（参考）

高危

  1. 流式分支 formatter 写错了 source_bu_type：养老险接口里硬编码了 "shouxian" 和"正在调用寿险红利接口...."文案
  2. 意图识别异常兜底偷偷走"养老险意图"：系统故障时用户会进入一键流程，运维难以察觉
  3. classify_user_intent 异常返回全空对象，不等于"其他"：会继续走 get_onekey_result，查空 query 的 table，可能误命中

中危

  1. 大模型返回的意图字符串没校验：模型返回带引号/前缀/全角空格都会判错
  2. get_onekey_result 的 except 抛新异常被翻译成 code=-1：fallback 到小安数据可能更合理
  3. 小安 chat 的重试只针对 token 失效，网络错误不重试：网络抖动一次直接 code=-1

低危

  1. 流式响应实际是"伪流式"，仍同步等 run 完成才发完成事件
  2. run_agent 的 finally 在异常时也只 set done_event，异常本身没传播给客户端
  3. request.buChannel.get(...) 潜在 NPE（buChannel 可能为 None）
